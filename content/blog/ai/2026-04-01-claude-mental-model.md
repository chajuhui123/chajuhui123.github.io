Claude Code의 확장 기능은 크게 두 가지 축으로 정리할 수 있습니다.

- **Knowledge** - Claude가 "알고 있는 것"
- **Workers** - Claude가 "따로 일하는 것"

그리고 이것들을 묶어서 공유하는 **Plugins**까지, 총 세 가지 개념을 정리해보았습니다.

---

## Knowledge

### 1. CLAUDE.md

항상 켜져 있는 규칙 파일입니다. 매 세션 시작 시 로드되고, 매 턴마다 컨텍스트를 소비합니다.

### 2. Skills

필요할 때만 불러오는 워크플로우입니다. 평소에는 이름과 설명만 참조하다가, 관련 작업이 생기면 그때 전체 내용을 로드합니다.

토큰 관점에서 보면 아래처럼 정리할 수 있어요.

- CLAUDE.md에 넣어야 할 내용을 Skills로 빼면 → **토큰 절약**
- Skills에 넣어야 할 내용을 CLAUDE.md에 넣으면 → **매 턴마다 낭비**

따라서 "매번 필요한 규칙인가, 아니면 가끔 필요한 워크플로우인가"를 기준으로 생각해볼 수 있습니다. 컨텍스트 윈도우는 유한하기 때문에, 해당 기준을 통해 토큰 사용 전략을 계획할 수 있습니다.

-> 컨텍스트 윈도우(Context Window): AI 언어 모델이 한 번에 처리할 수 있는 텍스트의 최대 양을 의미 (시스템 프롬프트, 히스토리, 인풋, 첨부 파일 및 문서를 모두 포함)

### 3. Commands

`/슬래시`로 직접 호출하는 워크플로우입니다. 마크다운 파일로 만들 수 있습니다.

Skills와 Commands는 커맨드로 만들 수 있지만, '실행 시점'에서 차이점이 있어요.

- **Commands** - 내가 호출할 때만 실행
- **Skills** - Claude가 상황을 판단해서 자동으로도 실행 가능. frontmatter, 자동 트리거, 참조 파일 등 더 많은 기능을 가짐

---

## Workers

### 1. Subagents

독립적으로 일하는 워커입니다. 작업을 시키고, 결과를 받으면 끝이 나고, 서로 소통하지 않는 것이 특징입니다.

### 2. Agent Teams

Subagents와 달리 서로 소통하는 워커입니다. 태스크 리스트를 공유하고 직접 메시지를 주고 받습니다.
둘의 구분 기준은 '의존성' 입니다. **서로 의존성이 있으면 Team, 없으면 Subagent** 라고 정리할 수 있습니다.

### Workers를 쓰는 이유?

메인 세션에서 파일 N개를 분석하면 그만큼 컨텍스트 윈도우가 금방 차버립니다. Subagent에게 시키면 별도 컨텍스트에서 처리하고 결과만 돌려주기 때문에, 메인 대화를 깔끔하게 유지할 수 있습니다.

Workers의 목적은 **컨텍스트 보호**라고 정리할 수 있습니다.

---

## Plugins

Skills와 Agents를 잘 만들어놨다면, **Plugin으로 묶어서 다른 프로젝트나 팀에 공유**할 수 있습니다. 라이브러리처럼 설치하면 공유할 수 있습니다.

---

## 실제로 어떻게 쓰고 있나요?

저는 업무에서 Commands 를 잘 사용하고 있어요. `/today`, `/capture`, `/learned`, `/park` 처럼 자동 트리거보다 필요할 때 직접 호출하는 방식이 저의 업무에는 효율적으로 동작했습니다.

일상적인 워크플로우는 Commands가 담당하고, Skills는 좀 더 맥락 의존적인 작업을 자동으로 처리하는 역할로 나눠 사용하고 있습니다. 또한 최근에는 프로젝트의 전반적인 패키지 업데이트를 진행하였는데, git worktree 를 생성하여 해당 트리에서는 Agent 가 스스로 패키지 업데이트 작업을 할 수 있게 워크 플로우를 세팅해준 후, 최종 목표인 '패키지 업데이트 성공' 을 수행하도록 한 경험도 있어요.

## 설계 결정 기준

| 기능        | 핵심 질문                                |
| ----------- | ---------------------------------------- |
| CLAUDE.md   | 매번 필요한 규칙인가?                    |
| Skills      | 특정 상황에서만 필요한 워크플로우인가?   |
| Commands    | 내가 직접 호출해서 즉시 실행할 작업인가? |
| Subagents   | 작업이 서로 독립적인가?                  |
| Agent Teams | 작업 간 의존성이 있는가?                 |
| Plugins     | 다른 프로젝트나 팀에 공유할 것인가?      |

CLAUDE.md와 Skills는 **토큰 소비 관점**으로 판단할 수 있다고 정리할 수 있습니다. (= 매 턴 소모되는 토큰 비용이 정당한가? 정당하다면 CLAUDE.md 로 세팅)

- 저는 현재 진행 중인 프로젝트에서 모노레포의 구조 및 디렉터리 규칙, 모델-훅-타입 등의 컨벤션, 특정 작업 때 실행하는 스크립트 규칙 등을 200줄 이하로 유지하며 관리하고 있습니다.
- 개인적으로 제일 많이 사용하고 있는 기능은 'Commands'입니다. 특히 Jira, Figma MCP 서버를 호출해서 사용하는 Commands 들을 업무에서 가장 자주 사용하고 있습니다.

Subagents는 **병렬성과 독립성**이 핵심이에요. 토큰이 충분히 남아 있어도, 작업이 독립적이라면 Subagent가 더 나은 선택이에요. Workers를 "컨텍스트 보호"라고 보는 것도 맞지만, 더 정확하게는 "독립적인 작업을 격리해서 병렬 처리하는 것 입니다.

- 독립적인 병렬 작업으로는 E2E 실행(시나리오 별로 나눠 동시에 진행), 대규모 파일 분석(파일 N개를 각 Subagnet 에게 나눠서 분석 후 결과 수집), 다국어 번역(언어별로 Subagent를 나눠 병렬 처리) 등을 사례로 들 수 있습니다.
- 의존성있는 협업 작업으로는 리팩토링(하나의 에어전트가 인터페이스를 설계하고, 다른 에이전트가 구현하며 서로 결과물을 참조), 풀스택 기능 구현(BE에이전트와 FE에이전트가 API 스펙을 공유하며 작업) 등이 있습니다.

이번 포스팅 정리를 통해, 클로드 코드가 어떻게 작동하고, 어떻게 토큰이 소비되고 있는지 그 과정을 이해할 수 있었습니다.
